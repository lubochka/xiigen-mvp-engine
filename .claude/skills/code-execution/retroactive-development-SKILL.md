# Retroactive Development Skill v1.0

## The Core Rule

**Bug in generated code → fix in the ENGINE, not the output.**

Generated code is disposable. The engine (AF stations, DNA guards, skill blocks, factory contracts) is the source of truth. Fixing a generated file without fixing the engine means the next generation reproduces the bug.

---

## When to Invoke

- Any bug found in a file generated by AF-1 Genesis or AF-7 RAG Update
- Any bug found in a service that was generated from a factory contract
- After any engine fix: to re-validate all flows that use the same task type

---

## Fix Propagation Protocol

### Step 1: Identify the generation source

```
Generated file has a bug
  ↓
Where was it generated?
  AF-1 Genesis → look at af1-genesis.ts (prompt template, archetype selection, DNA guard injection)
  AF-7 RAG Update → look at af7-rag-update.ts (compliance check, template selection)
  Manual code → fix directly (no propagation needed)
```

### Step 2: Fix the engine

| Bug type | Fix location |
|----------|-------------|
| DNA-1 violation (`class XModel`) | af1-genesis.ts: add DNA-1 guard to generation prompt; or add SK-DNA block |
| DNA-4 violation (no extends) | af1-genesis.ts: MicroserviceBase directive in prompt template |
| Wrong archetype selected | af4-rag-context.ts: archetype selector logic |
| Wrong skill blocks injected | af4-rag-context.ts: `selectSkillsForContext()` activation rules |
| Wrong prompt template | af3-prompt-library.ts: archetype template for this task type |
| fabricType wrong in contract | Factory contract F-XXXX: fix `fabricType` field value |

### Step 3: Re-validate all flows using the same task type

```bash
# Find all factory contracts using the same task type
grep -rn "taskType.*T-XXX" server/src/engine-contracts/

# Find all flows that reference this factory
grep -rn "factoryId.*F-XXXX" server/src/flow-templates/

# For each flow found: run its simulation test
npx jest --testPathPattern="flow-XXXX"
```

### Step 4: Verify the fix is generative, not cosmetic

**Test:** Delete the generated file. Re-run the generation. The new file must NOT contain the original violation.

If the new file still contains the violation → the engine fix is incomplete.

---

## The Iron Rule

**"Adapt test to accept wrong output" = session failure.**

If a test fails because the generated output violates DNA, the options are:
1. ✅ Fix the engine → regenerate → test passes with correct output
2. ❌ Change the test assertion to accept the violation → REJECT

Changing an assertion requires Luba's explicit approval. See `test-integrity-skill/SKILL.md` Rule FM-1.

---

## Blast Radius After Engine Fix

When any AF station, DNA guard, or prompt template changes:

```
Change in AF-4 (skill selector) → affects all flows that use AF-4
Change in AF-1 (genesis prompt) → affects all generated services
Change in af3-prompt-library   → affects all archetypes using updated template
Change in DNA guard injection  → affects all future generations
```

Check each item in the blast radius has a simulation test that still passes.

---

## Anti-Patterns

1. **"I fixed the generated file directly."** The file will be re-generated with the same bug. Fix the engine.
2. **"Only the one failing test needed fixing."** If the generation station is wrong, ALL flows using that station may produce the same violation. Check them all.
3. **"The DNA violation is minor, I'll note it."** All DNA violations compound. A DNA-1 class in one service becomes a pattern. Fix immediately.
4. **"I'll fix the engine after Phase N."** Retroactive fixes compound. The longer a generation bug exists, the more flows have wrong output. Fix now.
