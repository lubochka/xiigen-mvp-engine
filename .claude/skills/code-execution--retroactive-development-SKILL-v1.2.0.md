# Retroactive Development Skill v1.2

## The Core Rule

**Bug in generated code → fix in the ENGINE, not the output.**

Generated code is disposable. The engine (AF stations, DNA guards, skill blocks,
factory contracts) is the source of truth. Fixing a generated file without fixing the
engine means the next generation reproduces the bug.

---

## When to Invoke

- Any bug found in a file generated by AF-1 Genesis or AF-7 RAG Update
- Any bug found in a service generated from a factory contract
- After any engine fix: to re-validate all flows using the same task type
- **v1.1:** Any portability violation found during V9 gate or pre-commit check
- **NEW v1.2:** Any controller generated without `@UseGuards` / `@Roles` / `@Public()`
- **NEW v1.2:** Any service with direct SDK import (D-HIST-001 violation)

---

## Fix Propagation Protocol

### Step 1: Identify the generation source

```
Generated file has a bug
  ↓
Where was it generated?
  AF-1 Genesis       → look at af1-genesis.ts
  AF-7 RAG Update    → look at af7-rag-update.ts
  Manual code        → fix directly
  Portability violation (P-1..P-5) → see Portability Fix Propagation table
  Auth violation (A-1..A-3)        → see Auth Fix Propagation table (NEW v1.2)
  D-HIST-001 violation             → see Auth Fix Propagation table (NEW v1.2)
```

### Step 2: Fix the engine (DNA violations)

| Bug type | Fix location |
|----------|-------------|
| DNA-1 violation (`class XModel`) | af1-genesis.ts: add DNA-1 guard to generation prompt |
| DNA-4 violation (no extends) | af1-genesis.ts: MicroserviceBase directive in prompt template |
| Wrong archetype selected | af4-rag-context.ts: archetype selector logic |
| Wrong skill blocks injected | af4-rag-context.ts: `selectSkillsForContext()` |
| Wrong prompt template | af3-prompt-library.ts: archetype template for this task type |
| fabricType wrong in contract | Factory contract F-XXXX: fix `fabricType` field value |

### Step 3b: Auth Fix Propagation Table — NEW v1.2.0

Auth violations are distinct from portability violations. A portability fix corrects
metadata and tenantId propagation. An auth fix changes the security surface of HTTP
endpoints — adding guards that break existing unauthenticated callers and require
new tests. Classify all auth fixes as `AUTH_PROTECTION_ADDITION` per self-verification
v1.2.0, not `BACKWARD_COMPAT`.

| Violation | Detection | Controller/Service file fix | Genesis prompt fix |
|-----------|-----------|----------------------------|-------------------|
| **A-1: No @UseGuards on controller** | `grep -L "@UseGuards" *.controller.ts` | Add `@UseGuards(JwtAuthGuard, RolesGuard)` immediately before `@Controller(...)` decorator | Add to AF-1: *"Every @Controller MUST have @UseGuards(JwtAuthGuard, RolesGuard) declared before @Controller(). No controller may be generated without guards."* |
| **A-2: Route without @Roles or @Public()** | `diff <(grep -cE "@Get\|@Post\|@Put\|@Delete\|@Patch" ctrl.ts) <(grep -cE "@Roles\|@Public" ctrl.ts)` | Add `@Roles(ROLE.ROLE_STRING)` or `@Public()` above each route method lacking an auth declaration | Add to AF-1: *"Every route method (@Get, @Post, @Put, @Delete, @Patch) MUST have @Roles(...) or @Public() declared above it. No route may be generated without an auth declaration."* |
| **A-3: @Public() not in bypass-paths.registry.ts** | `grep -rn "@Public()" *.controller.ts` then check `server/src/auth/bypass-paths.registry.ts` | Add the route path string to `BYPASS_PATHS` array in `bypass-paths.registry.ts` | Add to AF-1: *"Any route decorated with @Public() MUST also appear in server/src/auth/bypass-paths.registry.ts. Emit a comment: // TODO: add to bypass-paths.registry.ts"* |
| **D-HIST-001: Direct SDK import** | `grep -rn "from '@elastic\|from '@anthropic\|from 'pg'\|from 'ioredis'" *.service.ts \| grep -v "fabrics/"` | Replace direct SDK import with the corresponding fabric interface injection (`IDatabaseService`, `IAIService`, `IQueueService`). If no fabric interface exists: raise as GAP before proceeding. | Add to AF-1: *"Services MUST NOT import @elastic, @anthropic, pg, or ioredis directly. Use fabric interface tokens: FABRIC_TOKENS.DATABASE_SERVICE, FABRIC_TOKENS.AI_SERVICE, FABRIC_TOKENS.QUEUE_SERVICE."* |

### Step 4: Auth fix verification protocol — NEW v1.2.0

Unlike portability fixes (metadata only), auth fixes change runtime behavior.
Use this 8-step protocol for every A-1..A-3 or D-HIST-001 fix:

```
1. Apply controller/service file fix (A-1..A-3 / D-HIST-001 from table above)

2. Run dna-compliance-guard v1.2.0 auth gate (A-1..A-3 + D-HIST-001):
   Expected: A-1 GUARDS >= CONTROLLERS
             A-2 AUTH_DECL >= ROUTES
             D-HIST-001 SDK imports = 0

3. Run V10 auth gate in flow-implementation-guide v1.3.0:
   Expected: authStatus = "AUTH_READY"

4. Run Rule 7 auth route tests (test-integrity v2.2.0):
   Expected: ≥1 test for 401 (no JWT), ≥1 test for 403 (wrong role)
   If tests do not exist: write them BEFORE proceeding
   Template: see test-integrity v2.2.0 Rule 7

5. For TIER-C target flows — Run Rule 8 cross-tenant JWT isolation test:
   Expected: R6 test passing (Tenant B token → Tenant C route → 401/403)
   If test does not exist: write it BEFORE proceeding
   Template: see test-integrity v2.2.0 Rule 8

6. Update STATE.json:
   "authStatus": "AUTH_READY",
   "authGaps": []

7. Apply genesis prompt fix (add directive to af3-prompt-library.ts)

8. Tag the commit: "auth: A-1/A-2/A-3 retroactive decoration — FLOW-NN now AUTH_READY"
   Or for D-HIST-001: "portability: D-HIST-001 SDK import removed — FLOW-NN fabric-first"

Classify change as AUTH_PROTECTION_ADDITION (not BACKWARD_COMPAT) per
self-verification-SKILL.md v1.2.0 Rule 2.
```

**AUTH_DEFERRED case:**
If auth infrastructure is absent (`auth.module.ts` not present), do not apply
A-1..A-3 fixes yet. Record `authStatus: AUTH_DEFERRED` and queue for Phase H
after AUTH-ROLES-GROUPS-PLAN-v3.0 Phases 1-4 deploy.

## Cross-Flow Auth Scope

When A-1 (no @UseGuards) is found in FLOW-N, scan the full fleet for the same violation:

```bash
# Find all controllers lacking @UseGuards
find server/src/engine/flows -name "*.controller.ts" -exec \
  grep -L "@UseGuards" {} \; 2>/dev/null | \
  sed 's|server/src/engine/flows/||' | cut -d'/' -f1 | sort -u
```

Each flow returned has the A-1 violation. Auth remediation for one flow does NOT
remediate others. Add each to the auth remediation queue.

```bash
# Find all services with D-HIST-001 violations (direct SDK imports)
grep -rl "from '@elastic\|from '@anthropic\|from 'pg'\|from 'ioredis'" \
  server/src/engine/flows --include="*.service.ts" | \
  grep -v "fabrics/implementations" | \
  sed 's|server/src/engine/flows/||' | cut -d'/' -f1 | sort -u
```

## The Iron Rule

**"Adapt test to accept wrong output" = session failure.**

This applies to auth violations. If a test passes because it doesn't check for
401/403 responses — that is not acceptable auth coverage. The auth behavior
must be asserted directly.
