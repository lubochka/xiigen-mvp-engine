# Retroactive Development Skill v1.1

## The Core Rule

**Bug in generated code → fix in the ENGINE, not the output.**

Generated code is disposable. The engine (AF stations, DNA guards, skill blocks,
factory contracts) is the source of truth. Fixing a generated file without fixing the
engine means the next generation reproduces the bug.

---

## When to Invoke

- Any bug found in a file generated by AF-1 Genesis or AF-7 RAG Update
- Any bug found in a service generated from a factory contract
- After any engine fix: to re-validate all flows that use the same task type
- **NEW v1.1:** Any portability violation found during V9 gate or pre-commit check

---

## Fix Propagation Protocol

### Step 1: Identify the generation source

```
Generated file has a bug
  ↓
Where was it generated?
  AF-1 Genesis → look at af1-genesis.ts
  AF-7 RAG Update → look at af7-rag-update.ts
  Manual code → fix directly
  Portability violation → see Portability Fix Propagation table below
```

### Step 2: Fix the engine (DNA violations)

| Bug type | Fix location |
|----------|-------------|
| DNA-1 violation (`class XModel`) | af1-genesis.ts: add DNA-1 guard to generation prompt |
| DNA-4 violation (no extends) | af1-genesis.ts: MicroserviceBase directive in prompt template |
| Wrong archetype selected | af4-rag-context.ts: archetype selector logic |
| Wrong skill blocks injected | af4-rag-context.ts: `selectSkillsForContext()` |
| Wrong prompt template | af3-prompt-library.ts: archetype template for this task type |
| fabricType wrong in contract | Factory contract F-XXXX: fix `fabricType` field value |

### Step 3: Portability Fix Propagation Table — NEW v1.1.0

Portability fixes are different from DNA fixes. DNA fixes target AF stations and
regenerate the file. Portability fixes target the service file directly AND the genesis
prompt directive — because portability requirements were not in the engine when these
flows were generated.

| Violation | Detection | Service file fix | Genesis prompt fix |
|-----------|-----------|------------------|--------------------|
| **P-1: ClsService import (GAP-01)** | `grep -rl "import.*ClsService" *.service.ts` | Replace every `this.cls.get(TENANT_CONTEXT_KEY)` with explicit `tenantId` from `EngineContract.input.tenantId`. Remove ClsService from constructor. | Add to AF-1 genesis prompt: *"CF-476: tenantId MUST come from the EngineContract input payload. The service constructor MUST NOT inject ClsService or nestjs-cls."* |
| **P-2: Missing @connectionType (GAP-16a)** | `grep -L "@connectionType" *.service.ts` | Add JSDoc block before class: `/** @connectionType FLOW_SCOPED @flowId FLOW-XX @portability MOBILE */` | Add to AF-1 genesis prompt: *"Every service file must have @connectionType FLOW_SCOPED JSDoc annotation before the class declaration."* |
| **P-3: Unscoped FREEDOM keys (GAP-09)** | `grep "freedom.get\|fromConfig" *.service.ts \| grep -v "flow[0-9]*_"` | Rename key from `'myParam'` to `'flow{NN}_myParam'`. Update GUIDE-B21 FREEDOM table. | Add to AF-1 genesis prompt: *"All FREEDOM config keys MUST use flow-scoped prefix flow{NN}_. Example: 'flow48_translation_enabled', not 'translation_enabled'."* |
| **P-4: Local interface clones (GAP-02)** | `grep "^interface IDb\|^interface IQueue\|^interface IFreedom" *.ts` | Delete local interface. Import from `fabrics/interfaces/` (NestJS_DI) or `@xiigen/engine-infra-interfaces` (PLAIN_TS). | Add to AF-1 genesis prompt: *"NEVER define local IDb, IQueue, or IFreedom interfaces. Import canonical fabric interfaces only."* |
| **P-5: Missing requiredCoInstalls (GAP-10)** | `grep "searchDocuments\|storeDocument" *.service.ts \| grep "xiigen-" \| grep -v "flow[0-9]*-"` | Add to `package.json`: `"xiigen": { "requiredCoInstalls": ["FLOW-N"] }`. Declare ALL hops. | Add note to genesis prompt: *"Cross-flow ES index reads require the source flow in xiigen.requiredCoInstalls."* |

### Step 4: Portability remediation verification protocol

Unlike DNA fixes (delete file → regenerate → verify), portability fixes use a different
verification protocol because they fix classification metadata and tenantId propagation,
not generation logic:

```
1. Apply service file fix (P-1..P-5 from table above)
2. Run dna-compliance-guard v1.1.0 portability gate (P-1..P-5 checks)
   Expected: all portability checks PASS
3. Run V9 in flow-implementation-guide:
   Expected: portabilityStatus = "MOBILE"
4. Update STATE.json: portabilityStatus = "MOBILE", portabilityGaps = []
5. Apply genesis prompt fix (add directive to af3-prompt-library.ts)
6. If flow is already ACTIVE:
   This is portability remediation, not re-generation.
   Do NOT delete and regenerate (service is behaviorally correct, only metadata was missing).
   Tag the commit: "portability: GAP-XX remediation — FLOW-NN now MOBILE"
7. Run FLOW-PORTABILITY-TEST-PROTOCOL Layer 1
   Expected: connectionType = FLOW_SCOPED on all service files
8. Classify change as PORTABILITY_REMEDIATION (not BACKWARD_COMPAT) per
   self-verification-SKILL.md v1.1.0 Rule 2
```

### Step 5: Re-validate all flows using the same task type (DNA only)

```bash
# Find all factory contracts using the same task type
grep -rn "taskType.*T-XXX" server/src/engine-contracts/

# Find all flows that reference this factory
grep -rn "factoryId.*F-XXXX" server/src/flow-templates/

# For each flow found: run its simulation test
npx jest --testPathPattern="flow-XXXX"
```

### Step 6: Verify the DNA fix is generative (DNA only)

Delete the generated file. Re-run the generation. The new file must NOT contain the
original violation. If the new file still contains the violation → engine fix is incomplete.

#### Step 6a — Prove genericity on ≥3 inputs of the same class (universal)

A cosmetic fix removes ONE bad output; a generative engine fix is correct for the
whole **class** of inputs. Prove it with a Jest-parameterized test that exercises
**at least three distinct inputs of the same type** through the fixed engine path:

```typescript
// it.each / test.each — ≥3 inputs of the same class, one engine fix must satisfy all
it.each([
  ['T-101', 'af1-genesis', /* expected token */],
  ['T-102', 'af1-genesis', /* expected token */],
  ['T-103', 'af1-genesis', /* expected token */],
])('engine fix is generative for %s/%s', async (taskType, station, expected) => {
  const out = await engine.generate(sampleContracts.getStationInput(taskType, station));
  expect(out.isSuccess).toBe(true);
  expect(out.data).not.toMatch(/<original-violation-pattern>/); // violation gone for every input
  // assert the corrected token is present, not merely that success is true
});
```

Rule: **one** engine change must make **all ≥3** cases pass. If only the original
input is fixed and a sibling input of the same class still reproduces the bug, the
fix is cosmetic — return to Step 2 and fix the generator, not the single output.

#### Step 6b — Regenerate AND commit the stale generated output (universal)

After the engine fix, any already-generated artifact produced by the old engine is
now **stale**. Regenerate every affected output and **commit the regenerated files
together with the engine fix in the SAME commit** — never ship an engine fix while
leaving old generated output checked in:

```bash
# 1. regenerate every artifact the fixed station/contract produces (blast-radius, see below)
# 2. confirm the violation is gone in the regenerated files
grep -rn "<original-violation-pattern>" server/src/engine/flows/  # expect: no hits
# 3. stage engine fix + regenerated outputs together
git add <engine-file> <regenerated-output-files...>
git commit -m "fix(engine): <BUG-ENGINE-NN> + regenerate affected output"
```

A commit that changes the engine but leaves the previously-generated files untouched
is an incomplete retroactive fix: the next reader still sees the buggy output.

## Cross-Flow Portability Scope

When P-1 (ClsService) is found in FLOW-N, scan the full fleet:

```bash
grep -rl "import.*ClsService\|from 'nestjs-cls'" server/src/engine/flows/ \
  --include="*.service.ts" | sed 's|server/src/engine/flows/||' | cut -d'/' -f1 | sort -u
```

Each flow returned has GAP-01. Portability remediation for one flow does NOT
remediate others. Add each to the portability remediation queue.

---

## The Iron Rule

**"Adapt test to accept wrong output" = session failure.**

This applies equally to portability violations. If a test passes because it
doesn't check ClsService behavior — that is not acceptable coverage for GAP-01.
The tenantId must be asserted in a concurrent execution test.
