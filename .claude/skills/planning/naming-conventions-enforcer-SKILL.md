# SK-430 — XIIGen Naming Conventions Enforcer

## Why This Skill Exists

`flow35-contracts.ts` tells a reader nothing. `meta-arbitration-engine-contracts.ts`
tells them exactly what engine capability lives in that file.

`"taskTypeId": "T521"` in a Jira comment or log tells a reader nothing.
`"taskTypeName": "BundleVersionGuard (T521)"` tells them what it does.

This skill enforces naming standards across:
1. File and directory names in the codebase
2. EngineContract schema fields
3. STATE.json entries
4. Jira comments generated by SK-429

---

## Rule 1 — File Names in engine-contracts/

### Pattern: `{domain-name}-contracts.ts` and `{domain-name}-seed-prompts.ts`

**NEVER:**
```
flow25-contracts.ts        ← flow number reveals nothing
flow35-seed-prompts.ts     ← what are these prompts for?
```

**ALWAYS:**
```
bfa-conflict-detection-contracts.ts        ← FLOW-25
self-developing-flow-engine-contracts.ts   ← FLOW-26
human-approval-workflow-contracts.ts       ← FLOW-27
queue-fabric-bootstrap-contracts.ts        ← FLOW-29
tenant-lifecycle-contracts.ts              ← FLOW-30
design-system-governance-contracts.ts      ← FLOW-31
code-generation-loop-contracts.ts          ← FLOW-33
meta-arbitration-engine-contracts.ts       ← FLOW-35
feature-registry-contracts.ts              ← FLOW-36
```

**Rule:** The flow number appears ONLY in the file header comment, never in the filename.
The filename must describe the DOMAIN, not the sequence number.

## Rule 3 — EngineContract Schema: flowId + flowName

Every `EngineContract` instance MUST include both:

```typescript
// BAD — machine-only, human can't understand context:
taskTypeId: 'T521',
// (no flowId, no flowName)

// GOOD — machine + human readable:
taskTypeId: 'T521',
taskTypeName: 'BundleVersionGuard',         // human-readable task type name
flowId: 'FLOW-33',                           // machine identifier for the flow
flowName: 'Code Generation Loop Bootstrap',  // human-readable flow name
```

**The `EngineContractParams` interface requires these four fields:**
- `taskTypeId` — machine ID (T521)
- `taskTypeName` — human name (BundleVersionGuard) — **must match the `name` field**
- `flowId` — flow machine ID (FLOW-33)
- `flowName` — flow human name (Code Generation Loop Bootstrap)

**Note:** `taskTypeName` is the same value as `name` in EngineContractParams.
`flowId` + `flowName` are new fields that trace the contract back to its flow context.

```json
// BAD:

  "flow_id": "FLOW-33",
  "current_phase": "phase-6"

  "flow_name": "Code Generation Loop Bootstrap",

Every STATE.json produced by planning or SK-427 MUST include `flow_name`.

const CODE_GENERATION_LOOP_QUALITY_GATES = [...]   // FLOW-33

```

## Rule 7 — Interface `I`-Prefix: DI Ports ONLY (G04 universal addition from llm_mvp_core)

The core convention is "interfaces are `I`-prefixed". In mvp that rule applies
**only to injectable service/port interfaces** — it must NOT be applied to
data-shape / DTO / event-payload interfaces.

```typescript
// ✅ DI / service-port interface — I-prefix REQUIRED
//    (mvp has ~143 of these: IApprovalService, IAnthropicClient, …)
export interface IApprovalService { approve(req: ApprovalRequest): Promise<DataProcessResult<Approval>>; }

// ✅ data-shape / DTO / event payload — NO I-prefix (mvp has 1000+ of these)
export interface AccessControlResult { allowed: boolean; reason: string; }
export interface AccountCreatedPayload { accountId: string; tenantId: string; }
```

**Why the carve-out matters:** porting the C# "all interfaces start with `I`"
rule naïvely would force a rename of a thousand DTOs and break every consumer.
The line is **role**, not syntax: *is this interface injected through DI as a
port?* → `I`. *Is it a data shape passed as a value?* → no `I`.

**Grep gate MUST target DI ports only** — never `grep "^export interface [^I]"`
across the whole tree (thousands of false positives on DTOs):

```bash
# ✅ check ONLY injected ports: interfaces referenced in a constructor / @Inject
grep -rnE "constructor\(.*: [A-Z][A-Za-z]+Service" server/src/   # un-I'd service injected → flag
grep -rn "@Inject(" server/src/                                  # cross-check tokens resolve to I-ports
# ❌ NEVER: grep "^export interface [^I]" server/src/  → floods on DTOs, useless
```

## Rule 8 — Test Names: `[Unit]_[Scenario]_[Expected]` (Jest describe/it)

A test name must describe the behavior, not the file. In Jest:

```typescript
// ✅ describe = unit; it = Method_Scenario_Expected (or the readable "should…when…")
describe('ApprovalService', () => {
  it('approve_DuplicateRequest_ReturnsFailure', () => { /* one metric */ });
  it('should reject approval when tenant quota exceeded', () => { /* one metric */ });
});

// ❌ it('works'), it('test 1'), it('approve') — describe nothing
```

The core form is `[Method]_[Scenario]_[Expected]`; mvp accepts either that token
form or the readable "should [behavior] when [condition]" — but never a vague
name. Test files are `*.spec.ts` with a domain name (never `helper.spec.ts`).

## Rule 9 — Commit & STATE Reference the Session/Deliverable

- **Commit** references the session and deliverable, in the existing mvp style:
  `P1: agent-constitution skill files` or `FLOW-XX Phase Y: <what> — <files>`.
  WHAT changed, not HOW; one logical change per commit. (See `phase-git-report`.)
- **STATE.json** carries a human `flow_name` / `phaseName` alongside the id (Rule
  4 above) so a reader knows *which capability* the phase is, not just `FLOW-33`.

## File Names — `kebab-case.ts` With a Domain Name (TS restatement of Rule 1/2)

mvp source files are `kebab-case.ts` and must name the **capability**, not a
number or a generic word: `bundle-version-guard.service.ts`, not `helper.ts`,
`utils.ts`, or `flow33-thing.ts`. (This is Rule 1/2 applied to `server/src` and
`client/src`; the flow number lives in the folder / header comment only.)
