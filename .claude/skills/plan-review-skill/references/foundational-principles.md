# Foundational Principles — Plan Gate Reference

This document defines 8 non-negotiable principles that apply to ALL plans, flows, and features in XIIGen. They are hard gates, not optional considerations. Every plan must explicitly answer all 8 gate questions or receive Luba sign-off that a principle is out of scope.

FC-12 (Principles Compliance) checks this document during every plan review. 12 failure classes total — FC-12 is the final structural gate.

---

## P1 — MULTI-TENANT BY DESIGN

**Rule:** Every feature is tenant-isolated by design with `tenantId` on every query, document, cache key, and event. Data isolation is structurally enforced.

**What this means for XIIGen:**
- ES index names include tenantId: `skills-{tenantId}`, `contracts-{tenantId}`
- AsyncLocalStorage carries tenantId — never passed as method parameter
- No method signature `foo(data, tenantId)` — tenantId comes from context
- Per-tenant config overrides via FREEDOM pattern

**Plan gate question:** Does every new index, table, cache key, and event include tenantId scope?

**Violation:** `await this.db.findAll({ type: 'skill' })` — no tenantId scope → Tenant A reads Tenant B data.

**Correct pattern:**
```typescript
const tenantId = this.clsService.get('tenantId');
await this.db.findAll({ type: 'skill', tenantId });
```

## P2 — SAFE CONFIG STORAGE

**Rule:** Configs stored via FREEDOM fabric pattern with per-tenant encryption, never hardcoded. Secrets excluded from logs and telemetry.

**What this means for XIIGen:**
- All API keys, model endpoints, and credentials go through `ISecretsService.getSecret(key, tenantId)`
- No `process.env.OPENAI_API_KEY` in business logic — only in provider bootstrap
- Per-tenant variable settings = FREEDOM config
- Invariant settings = MACHINE config
- Secrets never appear in OTel spans or log output

**Plan gate question:** Are all credentials retrieved via ISecretsService? Are per-tenant settings FREEDOM? Are invariant settings MACHINE?

**Violation:** `new Anthropic({ apiKey: process.env.OPENAI_API_KEY })` in a service class — hardcoded retrieval bypasses per-tenant override.

**Correct pattern:**
```typescript
const apiKey = await this.secrets.getSecret('OPENAI_API_KEY', tenantId);
const client = new Anthropic({ apiKey });
```

## P3 — ALWAYS IMPROVE PROMPTS (PromptOps)

**Rule:** Prompts are versioned assets with PromptAsset (id, version, content, evalScore, status). AF-9 Judge drives an improvement loop. Static prompt strings in code are architectural violations.

**What this means for XIIGen:**
- Every prompt is a `PromptAsset` record in the database
- Status lifecycle: DRAFT → ACTIVE → DEPRECATED
- Rollback = pointer swap to previous ACTIVE version (no code deploy)
- Multi-model critique: Claude → GPT-4o → Gemini for improvement cycles
- AF-9 quality threshold triggers improvement loop automatically

**Plan gate question:** Are new prompts `PromptAsset` records? Is the AF-9 improvement loop wired? Does rollback work as pointer swap?

**Violation:** `const prompt = \`Generate a service for ${taskType}...\`` hardcoded in af3-prompt-library.ts — no versioning, no improvement loop, no rollback.

**Correct pattern:**
```typescript
const asset = await this.ragService.getPromptAsset('af1-genesis-orchestration', tenantId);
// asset.content is the versioned, improvable prompt
```

## P4 — RAG STORAGE (Dual-Tier)

**Rule:** All generated knowledge stored in RAG with adaptive retrieval. TIER 1 = global ES production (port 9200). TIER 2 = local ES for Docker-based testing (port 19200).

**What this means for XIIGen:**
- New skill patterns → RAG, not hardcoded TypeScript arrays
- New knowledge generated by sessions → captured in RAG for future retrieval
- `docker-compose.test.yml` configures local ES on port 19200
- `npm run rag:seed` populates local ES before test suite runs
- Test suite uses TIER 2 (19200) — never production RAG

**Plan gate question:** Does new knowledge go into RAG (not hardcoded)? Is local RAG configured in docker-compose.test.yml (19200)? Does the test suite use local RAG?

**Violation:** Adding new skill block definitions as TypeScript constants in `af4-rag-context.ts` — should be RAG records, not code.

**Correct pattern:**
```typescript
// Knowledge retrieval via RAG, not hardcoded array
const skillBlocks = await this.ragService.getSkillBlocks(taskType, tenantId);
```

## P5 — ALWAYS IMPROVE (Self-Developing Engine)

**Rule:** Every fix makes the ENGINE better, not just the output. AF-11 captures what worked (`skillsActive`, `scoreDelta`). Failed patterns become negative exemplars. FLOW-26 drives the improvement loop.

**What this means for XIIGen:**
- Bug in generated code → fix in AF station or DNA guard, not just the output file
- AF-11 feedback record includes: `skillsActive: string[]`, `scoreDelta: number`, `patternId: string`
- Failed skill combinations → injected into RAG as negative exemplars
- Score drop ≥ 0.1 triggers improvement loop (not just logging)
- Promotion ladder: DRAFT → CANDIDATE → CORE (based on score history)

**Plan gate question:** Does the fix touch the ENGINE (AF station / skill / guard)? Does AF-11 capture skillsActive + scoreDelta? Do failed patterns become RAG exemplars?

**Violation:** Fixing generated code directly without updating the AF station that produced the error — next generation will reproduce the same bug.

**Correct pattern:**
```typescript
// In af11-feedback.ts
await this.feedback.record({
  taskId, tenantId,
  skillsActive: ['SK-DNA', 'SK-TEST'],
  scoreDelta: 0.15,
  outcome: 'PASS'
});
```

## P6 — PLANNING + ARBITRATING DECISIONS (BFA)

**Rule:** Every decision goes through Business Flow Arbiter with cross-flow impact analysis. New events cross-validated against FLOW-01–31. Semantic conflicts escalate via FLOW-25 Arbiter.

**What this means for XIIGen:**
- New task types must be validated against all 31 registered flows
- `bfaRegistration.events[]` required in every new EngineContract
- FLOW-25 Arbiter resolves semantic conflicts (two flows claiming same event)
- AF-2 planning gate checks cross-flow impact BEFORE generation begins
- DR-XXX decision record for every architectural decision

**Plan gate question:** Are new events cross-validated against FLOW-01–31? Does every new contract include bfaRegistration.events[]? Does FLOW-25 path exist for conflicts?

**Violation:** Adding a new `order.completed` event without checking that FLOW-08 already publishes it → silent conflict, double-processing.

**Correct pattern:**
```typescript
// In engine contract
bfaRegistration: {
  flowId: 'FLOW-32',
  events: ['order.validated'],
  conflictPolicy: 'ARBITER' // escalates to FLOW-25
}
```

## P7 — LOCAL TESTING

**Rule:** Unit + simulation + Docker testing for every new component. No cloud dependencies in test suite. Test sequence enforced: unit → sim → docker. `npm test` must pass locally.

**What this means for XIIGen:**
- Unit test: pure logic, no I/O, no providers
- Simulation test: `@nestjs/testing` with in-memory providers (DatabaseInMemory, QueueInMemory)
- Docker test: real containers via `docker-compose.test.yml` (ES:19200, PG:15432, LocalStack:14566)
- MAPE-K control loop: Monitor → Analyze → Plan → Execute (each testable in isolation)
- No test requires internet access or production credentials

**Plan gate question:** Is the unit→sim→docker path defined for every new component? Can `npm test` pass with only Docker (no internet)?

**Violation:** Test that calls `new Anthropic({ apiKey: process.env.OPENAI_API_KEY })` — requires real API key, cannot run in CI without secrets.

**Correct pattern:**
```typescript
// Simulation level: use mock AI provider
const module = await Test.createTestingModule({
  providers: [{ provide: IAiProvider, useClass: MockAiProvider }]
}).compile();
```

## P8 — OPEN SOURCE MODEL / COST + TRAINING

**Rule:** Track AI costs per tenant. Collect training data from external AI responses. Route to cheaper models when quality sufficient. Local model path (ollama) for dev cycles.

**What this means for XIIGen:**
- OTel token logging: `(tenantId, taskType, modelId, inputTokens, outputTokens)` per request
- AF-9 Judge verdicts captured as training exemplars (input/output/score/verdict)
- Contextual bandit routing (FLOW-29): score > 0.8 → cheap model; score < 0.6 → expensive model
- Ollama or mock provider for local dev — no API costs during development
- Simple tasks (classification, extraction) → cheap; complex tasks (generation, critique) → expensive

**Plan gate question:** Is token cost tracked per tenant + taskType? Are AF-9 verdicts captured as training data? Does a local model path exist for dev?

**Violation:** Every AF call routes to Claude Opus regardless of task complexity — cost not tracked, no routing logic, no training capture.

**Correct pattern:**
```typescript
// OTel cost tracking
span.setAttributes({
  'ai.tenant_id': tenantId,
  'ai.task_type': taskType,
  'ai.model_id': modelId,
  'ai.input_tokens': usage.inputTokens,
  'ai.output_tokens': usage.outputTokens
});
```
