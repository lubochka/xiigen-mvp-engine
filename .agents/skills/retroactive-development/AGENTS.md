# Retroactive Development — Agent Instructions

## Core Rule

**Bug in generated code → fix the ENGINE, not the output file.**

Generated `.ts` files are disposable. If you fix the generated file directly, the next generation recreates the bug.

---

## Decision Tree: Where to Fix

```
Bug found in a .ts file
  ↓
Was this file generated by AF-1 or AF-7?
  YES → fix the engine (see table below)
  NO  → fix the file directly (hand-written code)

Engine fix table:
  DNA-1/4 violation in generated code → af1-genesis.ts: prompt template or DNA guard injection
  DNA-9 missing in generated code     → af1-genesis.ts: CloudEvents directive
  Wrong archetype selected            → af4-rag-context.ts: selectSkillsForContext()
  fabricType wrong in contract        → factory contract F-XXXX: fix fabricType field
  Prompt template wrong               → af3-prompt-library.ts: update archetype template
```

---

## Propagation Checklist

After any engine fix:

```
☐ Identify which task types use the fixed station/guard/template
☐ Find all factory contracts using those task types
☐ Find all flows (FLOW-XX) referencing those factories
☐ Run simulation test for each affected flow
☐ Delete generated file → re-run generation → confirm no violation in new output
☐ Zero new failures in server test suite
```

---

## Generative Fix Test

```
1. Write failing tests (bug-to-tests-skill): L1 unit + L2 sim + L3 e2e
2. Apply engine fix
3. Delete generated file
4. Re-run generation
5. Run DNA audit on new generated file (dna-compliance-guard pre-commit gate)
6. Run all 3 levels → must PASS
```

---

## Red Flags

- "I edited the generated file" → Is the engine also fixed? If no → generation will recreate the bug.
- "Only one flow uses this task type" → Check all factory contracts. Task types are shared across flows.
- "The test passes after my fix" → Did you verify the fix is generative? Delete file → regenerate → confirm clean.
- "The assertion was wrong" → NEVER change assertion to accept wrong output. Fix the engine.

---

## Iron Rule

**"Adapt test to accept wrong output" = session failure.**
Assertion changes require Luba's explicit approval. If a test fails, fix the engine.
