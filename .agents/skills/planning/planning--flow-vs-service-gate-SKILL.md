# Flow vs Service Gate v1.0

## WHEN TO INVOKE

Step ⓪.5 — runs immediately after `implementation-mode-gate` (Step ⓪) in
every planning session. Before phase structure, before artifact numbers, before
any plan content.

**Only runs when `implementationMode` is `manual` or `hybrid`.** For pure
`af-pipeline` flows (FLOW-01 through FLOW-24): every service is generated by
the engine; this gate is N/A.

---

## WHAT THIS SKILL PREVENTS

Plans that correctly identify `implementationMode: hybrid` but then list
TypeScript services for capabilities that should be flow topologies. This was
the exact failure mode in FLOW-37 v1/v2 and FLOW-38 v1/v2: the mode was
correct but the capability classification was wrong.

---

## PROTOCOL (5 steps)

### Step 1: Extract capability list from the plan

From the plan's "What This Flow Builds" section, list every distinct capability:

```
Example from FLOW-37 v2 (incorrect):
  - CapabilityGapRepository (TypeScript service)
  - SkillIndexReader (TypeScript service)
  - PhaseRequirementsRegistry (TypeScript service)
  - SkillTestRunner (TypeScript service)
  - MockContextFactory (TypeScript service)
  - TraceFailureAnalyzer (TypeScript service)

Example from FLOW-37 revised (correct):
  - phase-requirements index (bootstrap — Phase A)
  - phase-capability-gate topology (FLOW via AF pipeline)
  - trace-failure-analyzer topology (FLOW via AF pipeline)
  - node-test-harness topology (FLOW via AF pipeline)
  - phase-skill-orchestrator topology (FLOW via AF pipeline)
  - mock-context-factory topology (FLOW via AF pipeline)
  - gap-signal-capture topology (FLOW via AF pipeline)
  - skill-lifecycle-coordinator topology (FLOW via AF pipeline)
```

### Step 2: Apply the four-question test to each capability

See `bootstrap-boundary-SKILL.md` for the full Q1-Q4 decision tree.
Apply it to every capability and record the decision.

### Step 3: Populate capabilityRouting[] in STATE.json

```json
{
  "capabilityRouting": [
    {
      "capability": "{name}",
      "decision": "FLOW | MANUAL | UTILITY",
      "flowId": "FLOW-XXA (if FLOW)",
      "phase": "A (if MANUAL)",
      "reasoning": "Q{N}=YES: {reason} | bootstrap-list: {entry}"
    }
  ]
}
```

### Step 4: Validate — challenge every MANUAL classification

For each capability marked MANUAL:
- Is it on the `bootstrap-boundary-SKILL.md` exhaustive list? → OK
- Is it not on the list? → `⛔ RECLASSIFY AS FLOW`

For each capability marked FLOW:
- Does it have a `flowId` assigned (e.g., FLOW-37A, FLOW-38B)?
- Does it have an archetype assigned?
- Is there a topology contract template in `contracts/topologies/`?

### Step 5: Revise phase structure if reclassifications occurred

If any capability was reclassified from MANUAL to FLOW:
- Phase A scope shrinks (remove TypeScript service from Phase A)
- Add a new phase for each FLOW capability (submit contract to AF pipeline)
- Update `implementationMode` if all phases are now af-pipeline

## GATE OUTPUTS

After this gate passes, the following must exist in STATE.json:

```json
{
  "capabilityRouting": [/* all capabilities classified */],
  "bootstrapPhaseAScope": [/* only MANUAL capabilities */],
  "afPipelineFlows": [/* list of FLOW capabilities with flowIds */],
  "gate05Result": "PASS"
}
```

Gate ⓪.5 fails if:
- Any MANUAL classification lacks a bootstrap-list justification
- Any FLOW classification lacks a flowId
- The total count of MANUAL capabilities exceeds the bootstrap-list count
  (typically ≤ 3: index, seed, ONE bootstrap skill)

## TEST SCENARIOS

Scenario A: FLOW-37 planning session
- Plan lists `CapabilityGapRepository.ts` as Phase B delivery
- Apply Q1: YES (gap scan quality improves via PromptOps) → reclassify FLOW
- Output: `capabilityRouting` adds `{ capability: "capability-gap-scanning", decision: "FLOW", flowId: "FLOW-37A" }`
- Phase B restructured: submit `phase-capability-gate` contract to AF pipeline

Scenario B: A utility helper that converts timestamps
- Q1: NO (output is always the same) → Q2: NO → Q3: NO → Q4: NO (not on list)
- Classification: UTILITY (TypeScript fine for genuinely stateless helpers)
- Note: this is rare — most capabilities that touch the engine state answer YES to Q1-Q3

## ANTI-PATTERNS

```
❌ "I'll just mark everything as FLOW to pass the gate"
   → Each FLOW decision requires a flowId and a topology contract template
   → Fake FLOWs are caught in Phase A gate when topology file is missing

❌ "Capability gap scanning is a script, not a flow"
   → Scripts are unregistered, unjudged, unimprovable
   → The capability-gap-scanner is FLOW-37A topology with validate.handler nodes

❌ Skipping this gate when implementationMode is hybrid
   → Hybrid mode is precisely when this gate matters most
   → Pure af-pipeline doesn't need it; pure manual is pre-engine; hybrid is the trap
```
