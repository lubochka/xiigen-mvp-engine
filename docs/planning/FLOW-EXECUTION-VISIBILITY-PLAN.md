## WHY THIS DOCUMENT EXISTS

FLOW-01 through FLOW-24 will be generated by FLOW-33's implement-family loop.
Before the first generation run begins, five gaps must be closed. None of them
require new flows, task types, or factory interfaces — they are method additions,
a new npm script, an ES index, and a FREEDOM config value.

Without these five items:
- A new arbiter registered in FLOW-01 Phase A has no way to be verified against
  previously accepted bundles from other flows
- The 11–12 E2E scenarios in each flow's test-matrix.json have no automated runner
- FLOW-33 can detect that regenerating FLOW-01 would affect FLOW-02 (T521 blast
  radius) but has no protocol for what Claude Code must do next
- Nobody can look at a single place and see which flows are GENERATED vs PROMOTED
  vs ACTIVE vs REGRESSED — this lives scattered across PHASE-COMPLETE files
- The promotion threshold that T521 enforces has no defined value

## ADDITION 1 — `replay_arbiter_on_bundle()` on F1345

### Problem

When FLOW-01 Phase A registers `routing::auth-security`, that arbiter becomes
active for all future generation rounds. But it has never been run against
previously accepted bundles from FLOW-25, FLOW-26, FLOW-29, FLOW-30, FLOW-31,
or FLOW-33. If the new arbiter would have failed on an already-promoted bundle,
there is currently no way to detect this — T521 only does forward blast-radius
graph traversal, not backward arbiter replay.

### Solution

Add one method to `IRegressionImpactService` (F1345):

```typescript
interface IRegressionImpactService {
  // Existing:
  computeBlastRadius(familyId: string, tenantId: string): Promise<BlastRadiusReport>

  // New (Addition 1):
  replayArbiterOnBundle(
    arbiterId: string,
    bundleId: string,
    tenantId: string
  ): Promise<ArbiterReplayResult>
}

interface ArbiterReplayResult {
  arbiterId: string
  bundleId: string
  score: number
  passed: boolean
  wouldHaveBlocked: boolean  // true if score < pass threshold
  notes: string[]
  replayedAt: ISO
}
```

`replayArbiterOnBundle` retrieves the accepted bundle from the bundle archive
(already stored by the generation loop after ACCEPT), runs the specified arbiter
against it, and returns whether it would have blocked the promotion.

### When to invoke

Every FLOW-01..24 Phase A gate must include:

```
□ For each newly registered arbiter:
  run replayArbiterOnBundle against all accepted bundles in the same archetype
  Assert: wouldHaveBlocked = false for all promoted bundles
  If wouldHaveBlocked = true: Claude Code escalates before proceeding to Phase B
```

### Where to add in FLOW-33

FLOW-33 Phase C (implement-family loop) — amend `IRegressionImpactService`
interface definition and stub. Implementation follows in Phase C generation.

## ADDITION 2 — `test:flow-matrix` npm script in FLOW-35 Phase F

### Problem

Every FLOW-01..24 plan includes `contracts/tests/FLOW-XX.test-matrix.json` with
11–12 E2E scenarios. Two scenarios per flow require `virtualClock: true`. There
is no npm script that reads this JSON and executes the scenarios. Without it,
every Phase E gate cannot run the scenarios automatically — a human must execute
them manually or write a custom script per flow.

### Solution

Add to `package.json` (server):

```json
"scripts": {
  "test:flow-matrix": "ts-node scripts/run-flow-matrix.ts"
}
```

`scripts/run-flow-matrix.ts`:

```typescript
// Usage:
// npm run test:flow-matrix -- --flow=FLOW-01
// npm run test:flow-matrix -- --flow=FLOW-01 --scenario=FLOW-01-T005
// npm run test:flow-matrix -- --flow=FLOW-01 --virtual-clock

import { FlowMatrixRunner } from '../src/engine/testing/flow-matrix-runner';

const args = parseArgs(process.argv.slice(2));
const runner = new FlowMatrixRunner({
  flowId: args.flow,
  scenarioFilter: args.scenario,
  virtualClockEnabled: args['virtual-clock'] ?? false,
  matrixPath: `contracts/tests/${args.flow}.test-matrix.json`,
});

const results = await runner.runAll();
printResults(results);
process.exit(results.failed.length > 0 ? 1 : 0);
```

`FlowMatrixRunner` reads the test-matrix JSON, executes each scenario against
the running service, and injects the virtual clock adapter for `virtualClock: true`
scenarios. It uses the existing E2E test infrastructure — it is a thin runner,
not a new test framework.

### Cross-flow edge tests (gap from analysis — restored here)

Per-flow tests verify internal scenarios. They do not verify the boundary between
flows. When FLOW-33 regenerates FLOW-01's T47, `npm run test:flow-matrix --flow=FLOW-02`
runs FLOW-02's internal scenarios — but none of those scenarios fire an actual
`UserOnboardingCompleted` event from FLOW-01's T49 and verify FLOW-02's T50 triggers
correctly. That test lives at the dependency edge, not inside either flow.

Edge test convention — one file per dependency arrow in the flow graph:

```
contracts/tests/cross-flow/
  FLOW-01_to_FLOW-02.edge.spec.ts   ← UserOnboardingCompleted → T50 triggers
  FLOW-02_to_FLOW-03.edge.spec.ts   ← OnboardingCompleted → FLOW-03 gate unlocks
  FLOW-02_to_FLOW-04.edge.spec.ts
  FLOW-02_to_FLOW-05.edge.spec.ts
  FLOW-02_to_FLOW-07.edge.spec.ts
  ...one file per arrow in the dependency graph
```

Each edge test: emits the upstream flow's terminal event with a shared `correlationId`,
asserts the downstream flow's entry task type fires within SLA, and asserts tenant
isolation (same event for tenantB must not trigger tenantA's downstream). Uses
`InMemoryQueueProvider` — no real queue needed.

Edge test stubs created in FLOW-35 Phase F (placeholders only). Stub pattern:

```typescript
describe('FLOW-01 → FLOW-02 boundary', () => {
  it('UserOnboardingCompleted triggers T50 ParallelProfileEnricher', async () => {
    // STUB — implemented when FLOW-02 generates T50
    expect(true).toBe(true);
  });
});
```

Stubs skipped by `test:cross-flow` automatically (detected by the `STUB` comment).
No false passes before the downstream flow implements the test.

### Two npm scripts to add (not one):

```json
"scripts": {
  "test:flow-matrix": "ts-node scripts/run-flow-matrix.ts",
  "test:cross-flow":  "jest contracts/tests/cross-flow/ --testPathPattern=edge.spec"
}
```

`test:cross-flow` skips stub-only files and runs only edge tests with real assertions.

### Every FLOW-01..24 Phase E gate must include:

```
□ npm run test:flow-matrix -- --flow=FLOW-XX
  Assert: all scenarios pass
  Assert: virtualClock scenarios ran with clock injection

□ npm run test:cross-flow (runs all non-stub edge tests)
  Assert: upstream edges to FLOW-XX pass
  Assert: downstream edges from FLOW-XX pass (or are stubs — acceptable)
  If an edge test that previously passed now fails: gate fails
```

### Where to add in FLOW-35

FLOW-35 Phase F (integration wiring) — add both npm scripts, FlowMatrixRunner class,
and all cross-flow edge test stub files for the full FLOW-01..24 dependency graph.
Same session as the watcher service update. Stubs only in Phase F — implementations
fill in during each downstream flow's Phase E.

## ADDITION 3 — `DRY_RUN` mode on T516 BootstrapOrchestrator

### Problem

When a FLOW-01..24 session begins, Phase A registers contracts, arbiters, CF rules,
and DAG JSON. If the plan bundle is structurally invalid (missing factory, wrong
archetype, CF collision), this is only discovered during generation — after spending
tokens on AI calls. A structural validation pass before any generation is the
right gate pattern.

### Solution

Add `executionMode` field to T516's execution context:

```typescript
interface BootstrapOrchestratorContext {
  flowId: string
  tenantId: string
  executionMode: 'FULL' | 'DRY_RUN'  // new field
  // ... existing fields
}
```

`DRY_RUN` behavior:
- Reads the plan bundle
- Validates all artifact IDs within known ranges (factory F1–F{max}, task type T1–T{max})
- Verifies all referenced arbiters exist in ArbiterRegistry
- Verifies all CF rules are in the flow's assigned range (no collision)
- Verifies all DAG JSON references valid task type IDs
- Returns `DryRunValidationReport` — no generation triggered, no AI calls made

```typescript
interface DryRunValidationReport {
  flowId: string
  valid: boolean
  errors: DryRunError[]  // if any — prevents Phase B from starting
  warnings: DryRunWarning[]
  artifactsValidated: number
}
```

### Every FLOW-01..24 Phase A gate must include:

```
□ npm run bootstrap:dry-run -- --flow=FLOW-XX
  Assert: DryRunValidationReport.valid = true
  Assert: DryRunValidationReport.errors.length = 0
  Gate fails if any errors — do not proceed to Phase B
```

### Where to add in FLOW-33

FLOW-33 Phase B (bootstrap infrastructure) — amend T516 genesis prompt to include
DRY_RUN execution mode. Iron rule addition: `IR-DRY-1: DRY_RUN mode MUST NOT
trigger any generation loop, AI call, or state mutation beyond validation`.

## ADDITION 4 — `flow-lifecycle` ES index in FLOW-35 Phase A

### Problem

When FLOW-33 runs its implement-family loop and generates FLOW-01, then later
regenerates FLOW-01 after a prompt improvement, there is no single place to look
up the generation history. PHASE-COMPLETE files contain this information but are
scattered across `sessions/FLOW-XX/` directories and are not queryable. At
operational scale with 24 flows generating and re-generating, someone needs to be
able to answer "was FLOW-01 previously promoted, by whom, and how many times has
it been regenerated?"

### Solution

Seed a `flow-lifecycle` Elasticsearch index in FLOW-35 Phase A. One document per flow:

```json
{
  "flowId": "FLOW-01",
  "tenantId": "tenant-uuid",
  "status": "GENERATED | TESTING | PROMOTED | ACTIVE | REGRESSED | DEPRECATED",
  "generationHistory": [
    {
      "runId": "uuid",
      "triggeredBy": "FLOW-33 Phase C",
      "arbitersUsed": 84,
      "roundsToAccept": 4,
      "promotedAt": "ISO",
      "promotedBy": "claude-code-session-XYZ",
      "promptVersions": {
        "T47": "v3",
        "T48": "v2",
        "T49": "v4"
      }
    }
  ],
  "currentGeneration": {
    "runId": "uuid",
    "status": "ACTIVE",
    "testMatrixLastRun": "ISO",
    "testMatrixResult": "PASS | FAIL | NOT_RUN",
    "lastRegressionCheck": "ISO",
    "blastRadiusLastComputed": "ISO"
  },
  "dependsOn": ["FLOW-25", "FLOW-33"],
  "downstreamFlows": ["FLOW-02"],
  "featureIds": ["FT-047", "FT-048", "FT-049"]
}
```

### Lifecycle state transitions

```
NOT_STARTED
  ↓ Phase A dry-run passes
GENERATED
  ↓ all Phase B–D gates pass
TESTING
  ↓ test:flow-matrix passes + manual review (if Phase C has human gate)
PROMOTED
  ↓ deployed to tenant runtime
ACTIVE
  ↓ T521 blast radius from upstream regeneration exceeds threshold
REGRESSED  ← triggers downstream test re-run protocol
  ↑ re-generation + re-promotion cycle back to PROMOTED
```

### Where to add in FLOW-35

FLOW-35 Phase A (foundations) — seed the `flow-lifecycle` index alongside
the `feature-registry` index. Add `IFlowLifecycleService` factory interface
(one new factory at F1491 — the first after FLOW-35's F1484–F1490 range).

Verify at FLOW-35 Phase A session start per SK-416.

**Two seed operations in Phase A (both mandatory):**

```
Seed 1 — Retroactive (infra flows already complete):
  FLOW-25, FLOW-27, FLOW-29, FLOW-30, FLOW-26, FLOW-31, FLOW-33
  → status: ACTIVE, generationHistory populated from PHASE-COMPLETE files
  → featureIds: [] (FLOW-36 Phase B fills these in retroactively)

Seed 2 — Forward (user flows not yet generated):
  FLOW-01 through FLOW-24, FLOW-34, FLOW-35, FLOW-36
  → status: NOT_STARTED
  → generationHistory: []
  → dependsOn: per the dependency graph in this document
```

Seed 2 is what enables the Flow Map tab in FLOW-36 Phase F to show all 24 user
flows as NOT_STARTED immediately — the index is queryable from day one, not
populated incrementally as flows execute.

### Operator visibility

The flow-lifecycle index drives the **Flow Map tab** in FLOW-36 Phase F dashboard
(per the recommendation in the prior analysis). The dashboard queries this index
and renders:

```
FLOW-01: ACTIVE   │ gen: v3 │ T47/T48/T49 │ FT-047,048,049 │ last run: 2h ago
FLOW-02: PROMOTED │ gen: v1 │ T50/T51/T52 │ FT-050,051,052 │ test: PASS
FLOW-03: TESTING  │ gen: v1 │ T53/T54     │ FT-053,054     │ test: IN_PROGRESS
...
```

## ADDITION 5 — `blast_radius_promotion_threshold` FREEDOM config

### Problem

FLOW-33's T521 RegressionImpactAnalyzer blocks promotion when blast radius
exceeds a threshold. The threshold is referenced throughout FLOW-33's arbiter
definitions but its value is never specified. Without a concrete value, the
first time T521 triggers, Claude Code doesn't know whether 2 downstream flows
affected = block or = warn.

### Solution

Add to FLOW-33's FREEDOM config seeding (Phase A):

```json
{
  "configKey": "blast_radius_promotion_threshold",
  "defaultValue": {
    "directDependencies": 0,
    "transitiveDependencies": 2,
    "note": "Block if any direct dependency affected. Block if >2 transitive."
  },
  "tunable": true,
  "description": "T521 RegressionImpactAnalyzer uses this to decide BLOCK vs WARN",
  "scope": "tenant"
}
```

The logic: a direct dependency being in the blast radius (FLOW-02 affected by
FLOW-01 regeneration) always blocks — it's the most critical case. Transitive
dependencies (FLOW-05 affected because FLOW-02 changed) block only if more than
2 are affected — smaller counts are acceptable with a warning and downstream
test re-run.

### After-regeneration protocol (what Claude Code does when blocked)

When T521 returns blast radius that exceeds threshold:

```
1. Status: FLOW-01 → REGRESSED in flow-lifecycle index
2. Claude Code receives EscalationBriefing with blast radius report
3. EscalationBriefing options:
   [A] Re-run downstream test suites:
       npm run test:flow-matrix -- --flow=FLOW-02
       If all pass: promote FLOW-01, update FLOW-02 status to ACTIVE
   [B] Scope the regeneration:
       Revert to previous prompt version for the specific task type that caused impact
       Re-run T521 on the scoped change
   [C] Escalate to Luba with full impact report
4. Chosen option logged to round-decisions.jsonl
5. If [A] passes: FLOW-01 → PROMOTED, downstream flows re-verified
```

This protocol is added to FLOW-33 Phase D (integration gate) — the session file
for Phase D must include these three options in the escalation response template.

### Where to add in FLOW-33

FLOW-33 Phase A (enum + contracts + arbiters) — seed `blast_radius_promotion_threshold`
config alongside other FREEDOM config values. FLOW-33 Phase D session file —
amend escalation response section to include the three-option protocol above.
