## Grammar

**G5 Kiosk / Single Action**

One-screen-at-a-time sequences: signup → pending → verify → onboarding. No
side navigation, no dashboard, no admin tables. Each screen has **one primary
action** only. G3 (Card List) and G6 (Dashboard) are both forbidden in this
flow; their presence is an Axis A BLOCK.

**Design reference set** (industry exemplars):
- Airbnb signup
- Linear onboarding
- Notion signup

Common shape: SSO buttons stacked above email form; single "Sign up" primary;
immediate onboarding after verify; first-load dashboard with personalised
suggestions (not a generic empty state).

## WHO × VERB per page × role

FLOW-01 has 6 client pages. Each page has one allowed role set and exactly one
primary action ("verb") per role. Axis B consults this table.

| Page | Route | Role | WHO | VERB |
|---|---|---|---|---|
| RegistrationPage | `/register` | anonymous | first-time visitor signing up for XIIGen Community | **Create account** (email + password OR SSO) |
| RegistrationPage | `/register` | tenant-user / tenant-admin / platform-* | already-signed-in user who navigated here by mistake | **Return to dashboard** (not signup form) |
| RegistrationPendingPage | `/register/pending-verification` | anonymous | newly-registered user waiting for verification email | **Check inbox** + secondary **Resend** + tertiary **Change email** |
| VerifyTokenPage | `/verify-email/:token` | anonymous | user who just clicked the link in their verification email | **Celebrate confirmation** → auto-redirect to onboarding within 2s |
| VerifyTokenPage (expired/used/invalid) | `/verify-email/:token` | anonymous | user whose link is no longer valid | **Recover via Resend** (route to ResendPage) |
| ResendPage | `/verify-email/resend` | anonymous | user who needs a new verification email | **Request resend** (rate-limited per FREEDOM config) |
| OnboardingPage | `/onboarding` | tenant-user | freshly-verified user setting up their workspace | **Complete 3-step wizard** (workspace name → first-flow tutorial → invite teammates) — Skip allowed |
| OnboardingPage | `/onboarding` | tenant-admin | admin after verify | **Complete wizard + Invite team members** |
| SsoPage | `/register/sso` | anonymous | user arriving from an SSO provider | **Confirm SSO identity** + one-time welcome |

**Anonymous-only entry rule**: any authenticated role navigating to `/register`
or `/register/pending-verification` must see a "you're already signed in" kiosk
with a route-to-dashboard button — NEVER the signup form. Axis B BLOCK if
authenticated role sees signup form.

## Shell (Axis A)

All 6 pages use the **Kiosk shell** — no sidebar, no top-bar admin chrome, no
tenant-switcher. The pages self-contain navigation via their primary action
and at most one secondary link.

Forbidden in any FLOW-01 page:
- Sidebar navigation
- Admin CRUD tables
- Tenant-switcher
- Role-switcher
- Page header with "Breadcrumbs > to > here"
- TopologyViewer / BusinessStateCard / AdminCrudPanel

Presence of any of these = Axis A BLOCK.

## Design signature (Axis F Layer 5)

From examination and functional spec:

- **Tone**: warm, welcoming, human. "Welcome to the community", not "User account created successfully (HTTP 200)"
- **Visual weight**: large central form; generous whitespace; single-column; primary CTA is a filled button (not outlined)
- **Motion**: subtle fade/slide between pages; 2-second celebratory animation on verify success
- **Iconography**: single large illustration or icon per page (not dense); email icon on pending; checkmark + confetti on verify success
- **Typography**: sans-serif system font; headings at 24-32px; body 14-16px; no all-caps headings
- **Colors**: primary brand blue for CTAs; success green for verify confirmation; error red used sparingly (uniform validation failure only)
- **Length**: no page renders scrollable content on desktop 1280px. If a page overflows, grammar is wrong (likely converted to G6 Dashboard by mistake).
- **Micro-copy**: verbs in the CTA (e.g. "Create account", "Resend email", "Continue"); no nouns ("Submit", "OK") as primary actions
- **Error tone**: uniform `VALIDATION_FAILURE` shape per FLOW-01-RAG-03 — no field name leaked in the error text

Presence of any of the following in the rendered PNG = Axis F Layer 5 concern:
- Admin console chrome
- Dashboard-style card grid
- CRUD table
- Engineering IDs (T47, T48, T49) or CF-rule numbers in user-facing copy
- Raw JSON error object
- Mixed-case namespace strings (`tenant_id`, `onboarding_materials`) visible

## Per-cell mandatory elements (Axis D source)

These are the minimum visible elements per page-state combination. If any is
missing, Axis D BLOCK.

### RegistrationPage — populated
- Email input field (labelled "Email" or i18n equivalent)
- Password input field (labelled "Password")
- "Sign up" / "Create account" primary button (filled)
- SSO alternatives section header (e.g. "or continue with")
- "Already have an account? Sign in" secondary link

### RegistrationPage — error
- All of above, PLUS:
- Uniform error message (no field name, per FLOW-01-RAG-03)

### RegistrationPendingPage — populated
- "Check your inbox" heading
- Masked email line ("We sent a link to v****@example.com")
- "Resend" secondary button
- "Change email" tertiary link

### VerifyTokenPage — populated (success)
- Success heading (e.g. "You're in!")
- Single illustration/icon
- Automatic redirect countdown or "Continue to onboarding" button

### VerifyTokenPage — error (expired/used/invalid)
- Clear error heading (not codes)
- Recovery CTA linking to ResendPage

### ResendPage — populated
- Email input (or display of current pending email)
- "Resend verification" button
- Rate-limit message when applicable (from FREEDOM `flow01_resend_rate_limit_minutes`)

### OnboardingPage — populated
- Workspace-name input (step 1 of 3)
- Progress indicator showing step 1/3
- Primary "Continue" CTA
- Optional "Skip" secondary

### SsoPage — populated
- Welcome message referencing SSO provider (e.g. "Welcome, signed in via Google")
- Single "Continue to XIIGen" CTA
