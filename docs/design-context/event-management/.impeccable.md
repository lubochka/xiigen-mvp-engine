## Grammar

**EventListPage → G2 CARD_LIST** — vertical stack of event cards; each card
renders title, status badge (ACTIVE / PROMOTED / DRAFT / ENDED), start date,
capacity label ("142 / 200"), and organiser id. Primary action per row:
**Promote** (only when status=ACTIVE) + **View**. Header carries the page
primary action: **+ Create Event** (filled blue button). Empty state: plain
dashed panel with "No events yet." + "Create your first event →" link. Error
state: red panel with "Failed to load events. Please refresh or try again."

**EventCreationPage → G5 Kiosk single-screen form (role-scoped)** — the
business-flow spec aspires to a 4-step wizard (Details → Tickets → Promotion →
Publish) BUT the implemented page is a **single-screen form with live
preview** (RUN-99 + RUN-115 signatures). The page is role-aware via
`RoleScopedView` (8 branches + fallback). The `event-organiser` branch renders
the creation form. Every other role branch renders a **CARD_LIST** of sample
events with a role-appropriate CTA (Register, Offer services, Sponsorship,
etc.). The tenant-admin branch renders a moderation table (approval queue).
The platform-admin branch renders a cross-tenant summary row (active total +
pending moderation).

**Design reference set** (industry exemplars):
- EventListPage: Eventbrite organiser dashboard, Luma event list
- EventCreationPage (organiser): Eventbrite create-event form with live
  preview, Luma create-event single-screen
- EventCreationPage (attendee / public): Meetup "upcoming events" list

Common shape across both pages: one primary filled-blue CTA per surface;
secondary actions are text links or outlined buttons; content in a white card
over a grey page; no admin chrome unless the role is admin-tier.

## Shell (Axis A)

Both pages render inside the **content area of the app shell** — no sidebar
or top-bar chrome is rendered by the pages themselves (the global app shell
may add one; the pages own only the white card area). The creation page
itself is a centered narrow card (`max-w-lg`); the list page is a
wider left-aligned column (`p-6` with a full-width header).

Forbidden in any FLOW-03 page surface (role-independent):
- Raw T-numbers or task-type IDs (`T59`, `T60`, `T61`, `T62`) anywhere in
  copy
- FREEDOM config keys (`flow03_max_events_per_organizer`,
  `flow03_max_attendees`, `flow03_promotion_channels`, etc.) visible as
  literal strings in UI copy — the user sees the **effect** (cap, waitlist,
  channel list), not the config key name
- Raw event names (`EventCreated`, `TargetAudienceIdentified`,
  `PromotionCampaignCompleted`, `EventPromoted`, `AttendeeRegistered`,
  `WaitlistJoined`, `EventAnalyzed`, `FeedsUpdated`, `NotificationsSent`)
  in user-facing copy — these are engineering identifiers
- Admin CRUD tables on a non-admin branch (attendee / organiser / freelancer)
- Role-switcher on any branch
- `knowledge_scope` / `connection_type` literals in copy
- `tenant` as a user-facing word on event-organiser / tenant-user / attendee /
  referral-user / freelancer / business-partner branches (tenant is an
  engineering concept; users see *their* content). The word is permitted on
  platform-admin branch only ("cross-tenant event overview") because that
  role's whole purpose is multi-tenant oversight.

Presence of any of these = Axis A BLOCK.

## Design signature (Axis F Layer 5)

From examination and the event-management business-flow spec:

- **Tone (event-organiser)**: practical and clear. "Create Event" — not
  "Instantiate Event Entity". "Organising as Acme Events" — not "Session
  tenant-user resolves to organiser-scope 'org-001'". Required fields marked
  with a single red asterisk (`*`) and a top-of-form "* Required" legend.
  Live preview caption reads "Live preview · What buyers see in the
  marketplace" — friendly, not engineering-flavoured.
- **Tone (attendee / visitor)**: welcoming and explanatory. "Sign in to create
  or manage events." / "You're viewing events as an attendee." / "You joined
  via a referral link. You'll earn rewards when you attend your first
  event." / "Discover events where you can offer your services." / "Discover
  events to sponsor or partner with." Each role gets a banner in a role-tinted
  pastel (blue / amber / purple / slate) — never harsh red except on error.
- **Tone (tenant-admin)**: operational. "Admin moderation view — review and
  approve pending events." / "Pending review" pill on queue rows.
- **Tone (platform-admin)**: compact summary (RUN-115 swap — two hero-tiles
  replaced with a single-line "Active events (all tenants) 284 · Pending
  moderation 12 needs review"). Cross-tenant language is allowed here and
  only here.
- **Visual weight**: both pages centred content over a white card on a grey
  page. EventListPage uses a full-width 3-card stack. EventCreationPage
  (organiser branch) uses a narrow single column (`max-w-lg`). The live
  preview sits BELOW the form, not beside it — current implementation is
  not a two-column layout despite the examination record calling for
  "right-panel live preview".
- **Motion**: none beyond default browser focus ring + hover colour shifts.
  Submitted form → success panel render is instant (no spinner between).
- **Iconography**: no hero illustrations. Status badges use tailwind colour
  coding (green ACTIVE, blue PROMOTED, gray DRAFT, red ENDED) + text. Live
  preview uses a small coloured dot (emerald) for the "Unlimited capacity"
  line; otherwise text-only.
- **Typography**: sans-serif system font; h1 is `text-2xl font-bold text-gray-900`
  (24-28px bold). Body is `text-sm` (14px). Micro-copy (required legend,
  "Live preview" caption) uses `text-xs` (11-12px) and `tracking-wider
  uppercase` for the preview eyebrow.
- **Colors (CTAs + state)**:
  - Primary CTA: filled `bg-blue-600 hover:bg-blue-700` (+ Create Event,
    Create Event, Browse events, Register →, Sign in to register)
  - Secondary: outlined button or text link
  - Success: green-50 / green-200 / green-700 panel for "Event created
    successfully."
  - Error: red-50 / red-200 / red-700 panel for "Event creation failed." or
    "Failed to load events."
  - Duplicate: yellow-50 / yellow-200 / yellow-700 panel for "An event with
    this key already exists."
  - Status badges on list cards:
    - ACTIVE → `bg-green-100 text-green-700`
    - PROMOTED → `bg-blue-100 text-blue-700`
    - DRAFT → `bg-gray-100 text-gray-600`
    - ENDED → `bg-red-100 text-red-700`
  - Role banners:
    - tenant-user attendee → `bg-blue-50 border-blue-200 text-blue-900`
    - referral-user → `bg-amber-50 border-amber-200 text-amber-700`
    - freelancer → `bg-purple-50 border-purple-200 text-purple-900`
    - business-partner → `bg-slate-50 border-slate-200 text-slate-700`
    - tenant-admin → `bg-orange-50 border-orange-200 text-orange-900`
    - platform-admin → `bg-red-50 border-red-200 text-red-900`
- **Length**: on desktop 1280×800, both pages render in a single viewport
  without scrolling in their default states. Creation-form + live-preview
  fits because the live preview compresses to ~300px. Event list scrolls
  only if > ~6 events (MOCK_EVENTS has 3 seeded).
- **Micro-copy**: verbs in CTAs ("Create Event", "Browse events", "Promote",
  "View", "Register →", "Sign in", "Offer services at this event", "Explore
  sponsorship"); never bare "Submit" / "OK" / "Save". Field placeholders
  use concrete examples ("e.g. Annual Tech Summit 2026", "e.g. 100").
- **Error tone**: field-level validation uses `text-red-600 text-xs` below
  the field with `border-red-400` on the input. Form-wide validation uses a
  `bg-red-50 rounded px-3 py-2` banner with role="alert". Error copy is
  neutral ("Please review your submission.") — NOT "Form invalid" nor "500
  error". Field errors do carry the field name in message text ("Event
  title is required", "Start date is required") — this is intentional;
  event-organisers benefit from knowing which field, unlike the FLOW-02
  questionnaire which deliberately scrubs field names from error copy.

Presence of any of the following in the rendered PNG = Axis F Layer 5 concern:
- Admin console chrome (sidebar, tenant-switcher, role-switcher) on
  non-admin branches
- Dashboard-style KPI-tile grid anywhere (RUN-115 explicitly replaced this
  with a single summary row on platform-admin — any PNG showing two or more
  hero tiles on the creation page = regression)
- CRUD table on a non-admin branch (only tenant-admin renders the
  `event-admin-queue`)
- Engineering identifiers in user-facing copy (T59/T60/T61/T62,
  `EventCreated`, `flow03_max_events_per_organizer`, `FLOW_SCOPED`)
- Raw JSON error object visible
- Mixed-case engineering namespaces visible (`xiigen-events`,
  `xiigen-event-registrations`, `xiigen-event-promotions`,
  `xiigen-event-analytics`, `freedom_configs`)
- The word **tenant** visible on any non-platform-admin branch
- Organiser ID typed into the form by the user (RUN-99 removed this input —
  the testid `organizer-id-input` is kept for test compatibility but is
  rendered visually-hidden with `opacity-0`; any PNG showing a labelled
  "Organiser ID" input field = RUN-99 regression)

prove and PNGs should **not** attempt to re-prove:
- `flow03_max_events_per_organizer` caps creation at the organiser's quota
  (4th attempt returns RATE_LIMIT_EXCEEDED server-side). UI cannot inspect
  the cap number directly — if an organiser ever sees a toast "Event limit
  reached" that's Axis D Layer-B material, but the spec currently does not
  surface this path visually.
- `flow03_max_attendees` caps capacity via `Math.min(capacity, freedomMax)`.
  UI cannot inspect the FREEDOM cap — it only ever shows the effective
  capacity.
