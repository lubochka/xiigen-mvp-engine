## Grammar

**G5 Kiosk → multi-step wizard**

Three-screen sequence: Questionnaire → Matching → Personalization. One question
group (or one outcome group) per screen. Step indicator on top ("2 of 3") plus
progress bar. **Skip** is secondary, always available. Each screen has **one
primary action** — "Continue", "See matches", "Finish setup" — never a bare
"Submit" or "OK". G3 (Card List) and G6 (Dashboard) are both forbidden in this
flow; their presence is an Axis A BLOCK.

**Design reference set** (industry exemplars):
- Typeform conversational forms
- LinkedIn profile setup wizard
- Notion onboarding

Common shape: one question/decision per screen; progress bar visible; Back
secondary, Next primary; skip path preserved at every step; final screen
celebrates completion and routes into the product.

## WHO × VERB per page × role

FLOW-02 has 3 client pages. Each page has one allowed role set and exactly one
primary action per role. Axis B consults this table.

| Page | Route | Role | WHO | VERB |
|---|---|---|---|---|
| QuestionnairePage | `/questionnaire` | tenant-user | freshly-onboarded developer building their profile | **Answer the next question group** (industry / stage / location / team) → **Continue** primary, **Skip** secondary |
| QuestionnairePage (duplicate) | `/questionnaire?duplicate=true` | tenant-user | user who just edited then re-submitted within the debounce window | **Wait** — see debounce-pending state, no re-submit allowed |
| QuestionnairePage (submitted) | `/questionnaire?mock=submitted` | tenant-user | user who completed the final answer | **See processing** — transient spinner with "Analysing your profile…" |
| MatchingPage (in-progress) | `/matching?userId=usr-matching-test` | tenant-user | user waiting for compatibility scoring to complete | **Wait for matches** — spinner + "Analysing compatibility across your network…" |
| MatchingPage (partial) | `/matching?userId=usr-partial-test` | tenant-user | user whose scoring hit B1 30s timeout (TIMEOUT-AS-SUCCESS-MODE) | **Review partial matches** — amber "Matches still being refined" notice + the 2 confirmed matches |
| MatchingPage (complete) | `/matching?userId=usr-complete-test` | tenant-user | user whose scoring finished in-window | **Review matches** — green "Matching complete" banner + connection count + sorted matches |
| PersonalizationPage (populated) | `/personalization?userId=usr-personalized-test` | tenant-user | user with feed ready | **Confirm feed preferences** → **PersonalizationCompleted** banner |
| PersonalizationPage (degraded) | `/personalization?userId=usr-degraded-feed-test` | tenant-user | user whose feed has degraded personalization (B2 partial) | **See degraded notice + feed still rendered** — no hard failure |

**Tenant-user-only entry rule**: platform-admin / platform-ops / tenant-admin
roles have **no** FLOW-02 UI. Any non-tenant-user navigating to
`/questionnaire`, `/matching`, or `/personalization` is outside scope (Axis B
BLOCK if admin chrome appears).

## Shell (Axis A)

All 3 pages use the **Kiosk shell** — no sidebar, no top-bar admin chrome, no
tenant-switcher, no role-switcher. The pages self-contain navigation via their
primary action (Continue / See matches / Finish) and at most one secondary link
(Skip / Back).

Forbidden in any FLOW-02 page:
- Sidebar navigation
- Admin CRUD tables
- Tenant-switcher
- Role-switcher
- Breadcrumb header ("Home > Profile > Enrichment")
- TopologyViewer / BusinessStateCard / AdminCrudPanel
- Raw T-numbers or task-type IDs (T50/T51/T52) anywhere in copy

Presence of any of these = Axis A BLOCK.

## Design signature (Axis F Layer 5)

From examination and the profile-enrichment business-flow spec:

- **Tone**: personal and warming. "Tell us about your work" — not "Capture
  user-profile attributes". "Here are people you'd click with" — not "Matched
  business profiles (count=3)".
- **Visual weight**: single-column wizard; one question group or one outcome
  group per screen; generous whitespace; primary CTA is a filled button.
- **Motion**: short cross-fade between wizard steps; the Matching spinner is
  the only long-running animation; on partial/complete transition, the amber
  "refining" notice gently replaces the spinner rather than slam-cutting.
- **Iconography**: at most one illustration or icon per screen. Questionnaire
  uses small inline field icons only. MatchingPage shows no decorative icons
  during partial/complete — just scores and names. PersonalizationPage is text
  + toggle chips only; no hero illustration.
- **Typography**: sans-serif system font; headings 24-32px; body 14-16px; no
  all-caps headings.
- **Colors**: primary brand blue for CTAs; success green for the "Matching
  complete" banner and the PersonalizationCompleted event chip; amber for the
  partial-results notice; error red used sparingly (uniform validation failure
  only — **never with the field name**).
- **Length**: no wizard page renders scrollable content on desktop 1280px. If
  a page overflows, grammar is wrong (likely mis-converted to a dashboard).
- **Micro-copy**: verbs in the CTA ("Continue", "See my matches", "Finish
  setup"); never "Submit" / "OK" / "Save" as primary. Secondary is always
  "Skip" or "Back".
- **Error tone**: uniform VALIDATION_FAILURE shape — error text must NOT
  contain the field name (`industry`, `stage`, `location`, `team`) nor the
  generic words `field` / `form` / `value` (this is explicitly asserted by
  Q-02 in profile-enrichment.spec.ts).

Presence of any of the following in the rendered PNG = Axis F Layer 5 concern:
- Admin console chrome (topbar admin chrome, sidebar)
- Dashboard-style card grid on the Matching screen (cards yes, dashboard-grid
  no)
- CRUD table
- Engineering identifiers in user-facing copy (T50, T51, T52, CF-rules)
- Raw JSON error object visible
- Mixed-case namespace strings visible (`xiigen-matching-profiles`,
  `FLOW02_MATCH_WEIGHT_INDUSTRY`, `knowledge_scope`, `connection_type`)
- The word **tenant** visible anywhere in tenant-user-facing copy (tenant is
  an engineering concept; the user sees only *their* content)

## Per-cell mandatory elements (Axis D source)

These are the minimum visible elements per page-state combination. If any is
missing, Axis D BLOCK.

### QuestionnairePage — empty (default)
- `questionnaire-form` form element (testid)
- Step indicator visible ("1 of 3" / "2 of 3" etc.)
- Progress bar (horizontal, filled to step fraction)
- First input group visible (industry / stage / location / team — only one per
  screen)
- **Continue** primary CTA (filled, verb, not "Submit")
- **Skip** secondary (text link, always present)
- Header copy in plain English — NOT "Capture user-profile attributes"

### QuestionnairePage — validation error (Q-02)
- All the above, PLUS:
- `questionnaire-validation-error` visible
- Error text MUST NOT contain `industry`, `stage`, `location`, `team`, `field`
- Error tone remains neutral; red pill or inline note, not a full-page banner

### QuestionnairePage — debounce pending (?duplicate=true)
- `debounce-pending` testid visible
- Micro-copy indicates wait ("You just updated this — we're applying changes…")
- Re-submit button is disabled or hidden

### QuestionnairePage — processing (?mock=submitted)
- `processing` testid visible
- Spinner + "Analysing your profile…" (or equivalent plain-English phrase)
- No error shown

### MatchingPage — in-progress (usr-matching-test)
- `page-matching` root visible
- `matching-in-progress` testid visible
- Spinner + "Analysing compatibility across your network…"
- Heading reads "Finding Your Matches" (or i18n equivalent)

### MatchingPage — partial (usr-partial-test)
- `page-matching` root visible
- `matching-partial` testid visible
- `partial-results-notice` testid visible (amber banner)
- 2 connection cards rendered (PARTIAL_CONNECTIONS: Acme Corp 94, Beta
  Ventures 87)
- Each card shows: name, score
- Heading reads "Your Matches"
- NO green "Matching complete" banner (partial ≠ complete)

### MatchingPage — complete (usr-complete-test)
- `page-matching` root visible
- `matching-complete` testid visible
- Green "Matching complete" banner visible
- `connection-count` testid visible with value "3"
- 3 connection cards rendered (MOCK_CONNECTIONS: Acme Corp 94, Beta Ventures
  87, Gamma Studio 81)
- Each card shows: name, score
- Heading reads "Your Matches"

### PersonalizationPage — populated (usr-personalized-test)
- `personalization-feed` testid visible
- ≥1 `feed-item` testid visible
- `personalization-completed-event` banner visible

### PersonalizationPage — degraded (usr-degraded-feed-test)
- `personalization-degraded` testid visible
- `personalization-feed` testid STILL visible (degraded but not absent)
- Copy acknowledges degraded state without raising error tone

## Machine invariants preserved (reference only — not a visual concern)

These are preserved at the server level and are **not** subject to visual
examination. Listed here only so the reviewer knows they exist and does not
attempt to cross-examine them via PNG.

- Event names: `BusinessProfileCreated`, `UserSegmentAssigned`,
  `LearningPathAssigned`, `BusinessMatchesFound`, `SuggestionsGenerated`,
  `MatchingPageRendered`, `PersonalizationCompleted`
- Task-type IDs: T50 (fan_in), T51 (convergence), T52 (broadcast)
- Dual-record write: PRIVATE `xiigen-business-profiles` + GLOBAL
  `xiigen-matching-profiles` (GLOBAL restricted to 4 match-safe fields)
- knowledge_scope literals: `GLOBAL` (matching), `PRIVATE` (profiles +
  matches + segments)
- connection_type: `FLOW_SCOPED`
- DNA-8 outbox: storeDocument before enqueue (all services)
- DNA-7 idempotency: sha256 key for scoringId
- TIMEOUT-AS-SUCCESS-MODE: partialResults=true is success, not failure

Any PNG that surfaces these identifiers (`T50`, `BusinessMatchesFound`,
`FLOW_SCOPED`, etc.) in user-facing copy = Axis F Layer 5 concern.
