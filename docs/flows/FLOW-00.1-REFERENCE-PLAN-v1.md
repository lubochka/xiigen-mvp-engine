## WHY THIS FLOW EXISTS

During FLOW-25 through FLOW-35, files and directories were named using
flow-number prefixes (`flow35-contracts.ts`, `engine/flows/flow35/`).
These names tell readers nothing about the domain or business purpose.

Additionally:
- `EngineContract` has no `flowId` or `flowName` fields — a contract cannot
  trace itself back to its flow context without reading the file's comments
- `STATE.json` has no `flow_name` — a reader sees "FLOW-33" but must look up
  what that means
- Jira comments generated by SK-429 lack business context: they report file
  counts and task IDs but not the business logic, the flow purpose, or how the
  code connects to tenant-facing capabilities
- Constants use `FLOW33_*` prefixes instead of domain-readable names

**This is not cosmetic.** When a new developer reads `flow35-contracts.ts`, they
cannot understand what they're looking at. When a Jira ticket says "registered T521",
they cannot understand what business capability was added or why.

**Business impact:** Naming is the first layer of documentation. If the names
are wrong, everything built on top of them is harder to understand, audit, and extend.

### A.3 — Create scripts/naming-lint.js

File: `server/scripts/naming-lint.js`

```javascript
/**
 * naming-lint.js — Enforces SK-430 naming conventions.
 *
 * Checks:
 * 1. engine-contracts/ files must NOT start with "flow" followed by digits
 * 2. engine/flows/ directories must NOT start with "flow" followed by digits
 * 3. *.ts files in engine-contracts/ must not contain FLOW{NN}_* constant names
 *
 * Run: node scripts/naming-lint.js
 * Exit 0 = pass, Exit 1 = violations found.
 *
 * Business purpose: Prevents regression to flow-number file naming.
 * Used by: CI gate, pre-commit hook, Phase A gate of every new flow.
 */

const fs = require('fs');
const path = require('path');

const violations = [];
const FLOW_NUM_PATTERN = /^flow\d+/;
const FLOW_CONST_PATTERN = /const FLOW\d+_/;

// Check 1: engine-contracts files
const contractsDir = path.join(__dirname, '../src/engine-contracts');
fs.readdirSync(contractsDir).forEach(file => {
  if (FLOW_NUM_PATTERN.test(file)) {
    violations.push(`VIOLATION [Rule 1] engine-contracts/${file} — uses flow-number prefix. Rename to domain name. See SK-430 Rule 1.`);
  }
  // Check for flow-number constants inside files
  const content = fs.readFileSync(path.join(contractsDir, file), 'utf8');
  const match = content.match(FLOW_CONST_PATTERN);
  if (match) {
    violations.push(`VIOLATION [Rule 6] engine-contracts/${file} — contains constant '${match[0]}...' — rename to domain prefix. See SK-430 Rule 6.`);
  }
});

// Check 2: engine/flows directories
const flowsDir = path.join(__dirname, '../src/engine/flows');
if (fs.existsSync(flowsDir)) {
  fs.readdirSync(flowsDir).forEach(dir => {
    if (FLOW_NUM_PATTERN.test(dir)) {
      violations.push(`VIOLATION [Rule 2] engine/flows/${dir}/ — uses flow-number directory name. Rename to domain name. See SK-430 Rule 2.`);
    }
  });
}

if (violations.length > 0) {
  console.error('\n❌ Naming lint FAILED:\n');
  violations.forEach(v => console.error('  ' + v));
  console.error(`\n${violations.length} violation(s). See SK-430 for fix instructions.\n`);
  process.exit(1);
} else {
  console.log('✅ Naming lint passed — all file and directory names comply with SK-430.');
  process.exit(0);
}
```
