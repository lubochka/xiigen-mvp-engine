## EXECUTIVE SUMMARY

FLOW-36 makes **Feature** a first-class artifact in XIIGen.

Today, a "feature" is implicit in code. It lives inside a platform adapter
(Figma plugin, Canva app) with no stable identity, no signal tracking, and no
systematic path to being ported to a new platform.

After FLOW-36:
- Every capability has a stable FT-ID that persists across platforms
- Business logic is canonical in XIIGen (Mode A); platforms get thin adapters (Mode B)
- Engine-internal capabilities are protected from accidental porting attempts
- Usage signals are tracked per FT per platform — Mode A and Mode B signals differ
- A porting decision is made by an arbiter — not a person's guess
- Platform simulation lets XIIGen test Mode B code without shipping it

## THE ARCHITECTURE THIS FLOW ESTABLISHES

### Two Modes of XIIGen Output

```
MODE A — Code runs on XIIGen
  Business logic, microservices, APIs
  Canonical. Fully observable. Trained.
  Signals: executionCount, successRate, avgCostPerRun, tenantAdoption

MODE B — Code ships to another platform
  Figma plugin, Canva app, Chrome extension, VS Code plugin
  Developed and tested INSIDE XIIGen
  Platform simulation handles the runtime boundary
  Business logic stays in Mode A; only the adapter is Mode B
  Signals: installs, activeUsers30d, likes, citations
```

### Three Feature Categories

```
CATEGORY 1 — Pure Mode A (engine-internal, portingCandidate=false)
  e.g. T516 BootstrapOrchestrator, T519 ImplementFamilyMetaLoop,
       T520 FiveArbiterConsensusGate, T521 RegressionImpactAnalyzer
  These ARE XIIGen. Porting them to Figma/Canva makes no architectural sense.
  PortingDecisionGate emits PortingProhibited immediately. No cost estimate run.

CATEGORY 2 — Mode A with porting potential (portingCandidate=true)
  e.g. T47 SSOAndEmailAuth, T50 ParallelProfileEnricher
  Business logic runs on XIIGen. A thin adapter could expose this to a platform.
  PortingDecisionGate evaluates Mode A signals + porting cost → APPROVE/DEFER/BLOCK.

CATEGORY 3 — Already Mode B (has existing adapter, portingCandidate=true)
  e.g. FT-001 DesignToCode (Figma plugin, implemented)
  Mode B signals (installs, likes) already being collected.
  PortingDecisionGate uses Mode B signals to decide additional platform ports.
```

### Feature as the Bridge Between Modes

```
FT-001: DesignToCode
  │
  ├── Canonical (Mode A):  DesignToCodeService in FLOW-31
  │   Fully tested, fully observable, generates training data
  │
  ├── Adapter (Mode B / Figma):  figma-adapter/FT-001/  ← implemented
  │   Thin. Calls Mode A via QUEUE FABRIC. Handles Figma API only.
  │
  └── Adapter (Mode B / Canva):  canva-adapter/FT-001/  ← generated by FLOW-34
      Same contract. Different platform API calls.
      Generated after PortingDecisionGate APPROVE.

FT-XXX: ImplementFamilyMetaLoop (T519)
  │
  ├── Canonical (Mode A):  XIIGen core generation engine
  │   portingCandidate=false
  │   portingCandidateReason="Engine-internal. Porting would expose generation
  │     internals to an external platform runtime. Architectural boundary violation."
  │
  └── [No adapters — PortingDecisionGate emits PortingProhibited immediately]
```

## PRINCIPLE ANSWERS (P1–P10)

### P1 — Multi-Tenant
Every FT record in the Feature Registry index carries `tenantId`.
A tenant's feature signals are isolated — tenant A's installs cannot
influence tenant B's porting decisions. RAG retrieval of feature manifests
is filtered by tenantId.

Isolation test: tenant A registers FT-001; tenant B cannot see it via
FeatureRegistry.getFeature(ftId, tenantB). Test in Phase A.

### P2 — Safe Configs
Platform API credentials (Figma OAuth, Canva API key) live in ISecretsService.
Signal ingestion webhook secrets in ISecretsService.
Porting cost thresholds, signal score thresholds: FREEDOM config (tenant-tunable).
portingCandidate classification: MACHINE (set at extraction time, never tenant-tunable).
No process.env.* for business values.

### P3 — Prompt Improvement
FeatureExtractor and PortingCostEstimator use versioned PromptAssets.
AF-9 verdict triggers PromptPatch proposals on low-quality extractions.
Promotion gate before new prompt version becomes active.

### P4 — Dual RAG
Global tier: Elasticsearch `feature-registry-{tenantId}` index (production).
Local tier: in-memory feature store in docker-compose.test.yml.
Tests use LOCAL tier only. Zero cloud credentials at any gate.

### P5 — Always Improve
Metrics per FeatureExtractor run: extraction_precision, ft_count,
boundary_accuracy, portingCandidate_accuracy, cost, latency, prompt_version.
Stored in FREEDOM layer.
Improvement cycle: low extraction_precision → PromptPatch → A/B test.
portingCandidate_accuracy: tracks how often auto-classification is overridden
by human (D-36-5 workflow) — feeds prompt improvement.

### P6 — Decision Nodes Arbitrated
Three arbitrated decision nodes:
1. FeatureExtractor — model arbiter (extraction quality + portingCandidate classification)
2. PortingCostEstimator — RAG arbiter (retrieves platform constraint docs)
3. PortingDecisionGate — governance arbiter (signal threshold + cost model +
   portingCandidate guard — PROHIBITED check is pre-arbiter, not arbiter-scored)
All registered in ArbiterRegistry. BFA rules in CF range (verified at session start).

### P7 — Local Testability
All 4 layers. docker-compose.test.yml adds: in-memory feature store,
mock platform signal webhook server, mock platform simulator.
npm test passes with ZERO cloud credentials.
Multi-tenant isolation test: in docker-local layer.
PROHIBITED path test: attempt porting of portingCandidate=false FT →
PortingProhibited emitted, no cost estimation run.

### P8 — Local Model Training
FeatureExtractor high-quality runs (extraction_precision ≥ 0.85) captured
as training data: (design_input, ft_manifest_output, score, taskType, tenantId).
portingCandidate classification included in output — teaches local model to
distinguish engine-internal from portable capabilities.
Per-tenant isolation enforced.
Local model endpoint in FREEDOM config. Graceful fallback to commercial.

### P9 — Mode C Event-First
All inter-service communication via QUEUE FABRIC only.
Event contracts in `contracts/events/FLOW-36/`.
Topology in `contracts/topologies/FLOW-36.topology.json`.
correlationId + tenantId + traceparent on every event.
No PII in any event payload (userId as reference only).
Compensation events for all non-terminal forward events.
New event: PortingProhibited (terminal, no compensation needed).

### P10 — Client-Side Architecture
Feature Matrix Dashboard: admin/developer sees FT × Platform matrix.
portingCandidate=false features shown with lock icon — no "Initiate Porting" button.
Port trigger workflow has client state map including PortingProhibitedScreen.
FlowStateSnapshot: `GET /flow/{flowId}/state` returns current porting step.
App-reopen: porting job resumes, no re-trigger.
Optimistic UI: "Initiate porting" button shows optimistic pending state,
confirmed by PortingJobStarted, rolled back by PortingJobRejected.
PortingProhibited: no optimistic state — server response is synchronous and immediate.
Payment/irreversible: N/A.
SDK: @xiigen/sdk-nestjs backend, @xiigen/sdk-react frontend.

**Signal score formula for MODE_B (FREEDOM config — tenant-tunable weights):**
```
signal_score = (
  w1 * normalize(installs) +
  w2 * normalize(activeUsers30d) +
  w3 * normalize(likes) +
  w4 * normalize(citations)
) * 100
Default weights: w1=0.3, w2=0.35, w3=0.2, w4=0.15
```

**Porting threshold formula for MODE_A (FREEDOM config — tenant-tunable weights):**
```
mode_a_porting_score = (
  w1 * normalize(tenantAdoption) +
  w2 * successRate +
  w3 * improvementVelocity
) * 100
Default weights: w1=0.5, w2=0.3, w3=0.2
Porting threshold default (both modes): score >= 60
```

## PORTING DECISION GATE — FULL LOGIC (updated v2)

```
PortingRequested event received
  │
  ▼
STEP 0 — portingCandidate guard (synchronous, pre-arbiter)
  if ftRecord.portingCandidate === false:
    emit PortingProhibited { ftId, reason: portingCandidateReason }
    ⛔ TERMINAL — pipeline ends, no cost estimation, no arbiter
    BFA CF rule: score-0 if this step is skipped

  if ftRecord.portingCandidate === true:
    continue to STEP 1
  │
  ▼
STEP 1 — incompatibility check
  if targetPlatform in ftRecord.platformIncompatibilities:
    emit PortingBlocked { ftId, targetPlatform, reason }
    ⛔ TERMINAL — permanent, no retry
  │
  ▼
STEP 2 — PortingCostEstimator (T-[+2])
  RAG: platform constraint docs + historical porting effort
  Output: cost_estimate, effort_days
  │
  ▼
STEP 3 — Signal threshold evaluation
  Determine signal mode from ftRecord:
    if platforms[] is empty:        use FeatureSignals (MODE_A formula)
    if platforms[] has adapters:    use ModeBSignals from most-adopted platform
  Compute porting score
  │
  ▼
STEP 4 — GOVERNANCE ARBITER decision
  APPROVE if:
    porting_score >= threshold (FREEDOM)
    AND cost_estimate <= max_porting_cost (FREEDOM)
    AND targetPlatform not deprecated

  DEFER if:
    porting_score < threshold
    emit PortingDeferred { ftId, estimatedReadyDate }

  BLOCK if:
    cost_estimate > max_porting_cost AND no path to reduce cost
    emit PortingBlocked { ftId, reason: "cost_ceiling" }
```

**New events in `contracts/events/FLOW-36/` (v2 adds):**
```
PortingProhibited.schema.json
  source: "server" | trigger: portingCandidate=false guard
  data: { ftId, tenantId, reason: string }
  TERMINAL — no compensation. prohibition is permanent until human override (D-36-5).
```

### PHASE B — Feature Extraction Pipeline

**Objective:** Implement FeatureExtractor (T-[+0]) end-to-end.
Given a codebase or design input, extract FT entries with portingCandidate
classification included.

**portingCandidate classification rules (in genesis prompt SK-[+0]):**
```
portingCandidate = false when ANY of:
  - The capability IS the generation engine or a sub-component of it
    (arbiters, consensus gates, bootstrap orchestration, regression analysis)
  - The capability requires access to XIIGen's internal state to function
    (BFA rules engine, training trace writer, prompt evolution)
  - Exporting this capability would expose XIIGen's architectural internals
    to an untrusted third-party platform runtime

portingCandidate = true when:
  - The capability is a well-bounded service with defined inputs/outputs
    that could run in a reduced-privilege platform sandbox
  - The business logic can be wrapped by a thin adapter without
    exposing internal state
  - The capability maps to a recognized platform plugin pattern
    (design tools, productivity tools, automation)
```

**Key implementation decisions:**

  const ft = await this.registry.getFeature(ftId, tenantId);

score:  decision_accuracy
